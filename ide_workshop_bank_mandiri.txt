      - Buat Semantic View dengan metrics relevan:
        * Total Dana Pihak Ketiga (DPK)

      -> Agent ambil SOP dari Cortex Search + data dari Cortex Analyst
   
   D) SNOWFLAKE INTELLIGENCE:
      - Publish agent sebagai aplikasi yang bisa dipakai business user

DAY 1 - SESSION 2: Data Protection Policies
================================================================================

TEMA: "Proteksi Data Nasabah Sesuai Regulasi OJK & UU PDP"

>> IDE SKENARIO:
   Simulasi governance di bank dengan multiple roles dan branch-level access.
   
   A) SETUP ROLES (Banking Hierarchy):
      - BANK_ADMIN (setara IT Security)
      - COMPLIANCE_OFFICER (audit & kepatuhan)
      - BRANCH_MANAGER_JKT (cabang Jakarta)
      - BRANCH_MANAGER_SBY (cabang Surabaya)  
      - TELLER_ROLE (frontliner)
      - DATA_ANALYST (tim data/BI)
      - RISK_ANALYST (tim risiko)
   
   B) DATA MASKING - Skenario Realistis:
      - Kolom NIK/KTP: masked jadi "327XXXXXXXXX01" untuk analyst
      - Kolom No Rekening: partial mask "123-XX-XXXX-789"
      - Kolom Nama Nasabah: full mask untuk teller, visible untuk compliance
      - Kolom Saldo: range masking (tampilkan range, bukan exact number)
        untuk analyst, exact untuk branch manager
      - Conditional masking: COMPLIANCE_OFFICER bisa lihat semua,
        DATA_ANALYST hanya lihat data ter-mask
      - Tag-based masking: tag "PII" otomatis apply masking policy
   
   C) ROW ACCESS POLICY - Branch Segregation:
      - BRANCH_MANAGER_JKT hanya lihat nasabah cabang Jakarta
      - BRANCH_MANAGER_SBY hanya lihat nasabah cabang Surabaya
      - Mapping table: ROLE <-> BRANCH_CODE <-> REGION
      - Demo: login sebagai role berbeda, data yang muncul berbeda
      - Skenario: "Setiap Regional Head hanya bisa akses data region-nya"
   
   D) PROJECTION POLICY - Anti Bulk Extraction:
      - Kolom NIK tidak bisa di-SELECT tanpa WHERE clause
      - Analyst HARUS specify filter (misal WHERE branch = 'Jakarta')
      - Mencegah: SELECT NIK, NAMA FROM NASABAH (bulk download semua data)
      - Mengizinkan: SELECT NIK, NAMA FROM NASABAH WHERE ACCOUNT_ID = 'xxx'
      - Ini sangat relevan untuk compliance UU Perlindungan Data Pribadi (PDP)

>> AHA MOMENT:
   Demo live: switch antar role, peserta langsung lihat perbedaan data
   yang tampil. "Satu tabel, tapi setiap role melihat data yang berbeda."
   Ini langsung menjawab kebutuhan segregation of duties di perbankan.

>> KONEKSI KE REGULASI:
   - UU No. 27 Tahun 2022 tentang Perlindungan Data Pribadi (UU PDP)
   - POJK tentang Tata Kelola TI Bank Umum
   - Prinsip Least Privilege & Need-to-Know basis

DAY 2 - SESSION 3: End-to-End Machine Learning
================================================================================

TEMA: "Credit Scoring Model untuk Kredit UMKM"

>> IDE SKENARIO:
   Build credit scoring model end-to-end, sangat relevan karena Bank Mandiri
   fokus ke lending UMKM via Livin' Merchant (cash-flow based lending).

   A) DATA PREPARATION (Snowflake Notebook):
      - Dataset sintetis: data pengajuan kredit UMKM
        * Fitur: omzet_bulanan, lama_usaha, jumlah_karyawan, sektor_usaha,
          lokasi, riwayat_kredit, jumlah_transaksi_digital, saldo_rata2,
          frekuensi_transaksi, rasio_tabungan
        * Target: STATUS_KREDIT (Lancar/DPK/Kurang Lancar/Diragukan/Macet)
          atau binary: DEFAULT / NON-DEFAULT
      - EDA dengan visualisasi di Notebook
      - Data cleaning dan transformasi dengan Snowpark Python
   
   B) FEATURE ENGINEERING & FEATURE STORE:
      - Buat features yang relevan untuk perbankan:
        * avg_monthly_transaction_amount (rata-rata transaksi bulanan)
        * transaction_frequency_trend (tren frekuensi 3 bulan terakhir)
        * saving_ratio (rasio tabungan vs pengeluaran)
        * digital_engagement_score (seberapa aktif di channel digital)
        * credit_utilization_ratio
      - Register ke Snowflake Feature Store
      - Tunjukkan feature versioning & discovery
   
   C) MODEL TRAINING:
      - Train XGBoost classifier untuk credit scoring
      - Hyperparameter tuning
      - Model evaluation: AUC-ROC, precision-recall (penting untuk banking:
        cost of false negative >> cost of false positive)
      - SHAP values untuk explainability -> WAJIB untuk regulasi OJK
        (model harus bisa dijelaskan kenapa approve/reject)
   
   D) MODEL REGISTRY & DEPLOYMENT:
      - Log model ke Snowflake Model Registry
      - Versioning: v1 (baseline), v2 (tuned)
      - Batch inference: scoring semua aplikasi kredit pending
      - Real-time inference via REST API: scoring saat nasabah apply online
   
   E) ML OBSERVABILITY & LINEAGE:
      - Monitor model drift: distribusi fitur berubah seiring waktu
      - Alerting: jika accuracy turun di bawah threshold
      - ML Lineage: trace dari source data -> features -> dataset -> model
      - Ini KRUSIAL untuk audit OJK: "Dari mana data model ini berasal?"

>> AHA MOMENT:
   Peserta bisa build credit scoring model yang production-ready dalam
   satu session, lengkap dengan explainability (SHAP) dan lineage.
   Bandingkan dengan proses tradisional yang butuh berbulan-bulan.

>> KONEKSI KE BANK MANDIRI:
   - Mereka sudah pakai AI untuk fraud detection (85% reduction)
   - Credit scoring adalah natural next step
   - Livin' Merchant menghasilkan data UMKM real-time -> perfect untuk
     cash-flow based lending model

REKOMENDASI DATASET SINTETIS YANG PERLU DISIAPKAN
================================================================================

1. DIM_NASABAH (10,000 rows)
   - nasabah_id, nama, nik, no_hp, email, tanggal_lahir, jenis_kelamin
   - segmen (Retail/Priority/Private/SME/Corporate)
   - cabang_id, tanggal_buka_rekening, status_aktif

2. DIM_CABANG (100 rows)
   - cabang_id, nama_cabang, kota, provinsi, region
   - tipe (KC/KCP/KK), jumlah_karyawan

3. FACT_TRANSAKSI (500,000 rows)
   - transaksi_id, nasabah_id, tanggal, jenis_transaksi
   - (Transfer/Tarik Tunai/Setor/Pembayaran/Top Up)
   - jumlah, channel (Livin/ATM/Teller/Kopra/EDC)

4. FACT_KREDIT (5,000 rows)
   - kredit_id, nasabah_id, jenis_kredit, jumlah_pinjaman
   - tenor, suku_bunga, tanggal_pencairan, status_kolektibilitas
   - (1-Lancar, 2-DPK, 3-Kurang Lancar, 4-Diragukan, 5-Macet)

5. FACT_SIMPANAN (15,000 rows)
   - simpanan_id, nasabah_id, jenis (Tabungan/Deposito/Giro)
   - saldo, suku_bunga, tanggal_buka

6. DOC_SOP_BANK (5-10 PDF files)
   - SOP Pembukaan Rekening
   - Panduan KYC
   - Kebijakan Kredit UMKM
   - Prosedur Restrukturisasi
   - Panduan Anti Money Laundering
